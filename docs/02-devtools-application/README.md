# DevTools - Application

## Вступ

Панель Application в Chrome DevTools надає доступ до всіх даних, що зберігаються браузером: Cookies, LocalStorage, SessionStorage, IndexedDB, Cache Storage та інші. Для Senior QA Automation Engineer розуміння Application панелі критично важливе, оскільки багато сучасних веб-додатків використовують клієнтське зберігання для кешування даних, збереження стану користувача та токенів авторизації. Вміння керувати цими даними дозволяє тестувати різні сценарії, очищати стан перед тестами та перевіряти коректність збереження даних.

**Важливо:** Всі приклади в цій документації працюють на реальних сайтах:
- **example.com**: https://example.com - для базових тестів
- **JSONPlaceholder**: https://jsonplaceholder.typicode.com/ - для тестів з API та збереженням даних

Всі приклади можна виконати одразу в DevTools Console або в Cypress тестах.

## Теоретичне пояснення

Панель Application в Chrome DevTools організована в лівій панелі з різними розділами: Storage (Cookies, Local Storage, Session Storage), Cache Storage, IndexedDB, Web SQL (застаріле) та інші. Кожен розділ дозволяє переглядати, редагувати, додавати та видаляти дані, що зберігаються браузером.

Для автоматизованого тестування Application панель дозволяє очищати всі збережені дані перед запуском тестів (щоб забезпечити чистий стан), перевіряти, чи правильно зберігаються дані після певних дій, модифікувати збережені дані для тестування різних сценаріїв, та перевіряти безпеку збереження чутливих даних. Це особливо важливо при тестуванні SPA, де стан додатку часто зберігається локально.

## Основні концепції

### Cookies

- **Читання**: Перегляд всіх cookies для поточного домену
- **Видалення**: Видалення окремих cookies або всіх одночасно
- **Модифікація**: Зміна значень, expiration date, domain, path
- **Атрибути**: HttpOnly, Secure, SameSite

### LocalStorage та SessionStorage

- **LocalStorage**: Дані зберігаються постійно до явного видалення
- **SessionStorage**: Дані зберігаються тільки протягом сесії браузера
- **API**: `localStorage.setItem()`, `localStorage.getItem()`, `localStorage.removeItem()`
- **Обмеження**: Зазвичай 5-10MB на домен

### Очищення кешу перед тестами

- **Clear storage**: Очищення всіх даних одним кліком
- **Selective clearing**: Видалення тільки певних типів даних
- **Application state**: Очищення стану додатку для чистих тестів

### Перевірка збережених даних

- **Data validation**: Перевірка структури та валідності даних
- **Security check**: Перевірка, що чутливі дані не зберігаються небезпечно
- **State verification**: Перевірка, що стан додатку правильно зберігається

## Поради для Senior QA Engineer

### Best Practices

1. **Очищайте storage перед тестами**: Завжди очищайте LocalStorage/SessionStorage перед кожним тестом для забезпечення ізольованості
2. **Перевіряйте структуру даних**: Перевіряйте не тільки наявність даних, але й їх структуру та валідність
3. **Тестуйте expiration**: Перевіряйте, як додаток обробляє expired cookies та дані
4. **Перевіряйте безпеку**: Переконайтеся, що чутливі дані (паролі, токени) не зберігаються небезпечно
5. **Використовуйте правильні методи**: Використовуйте `localStorage` для постійних даних, `sessionStorage` для тимчасових

### Advanced техніки

1. **Bulk operations**: Використовуйте скрипти для масового додавання/видалення даних
2. **Data export/import**: Експортуйте дані для відтворення стану в інших браузерах
3. **Storage events**: Відстежуйте події зміни storage для тестування синхронізації
4. **Quota management**: Тестуйте поведінку додатку при досягненні лімітів storage
5. **Cross-tab communication**: Тестуйте синхронізацію даних між вкладками

### Налагодження

1. **Перевірка значень**: Завжди перевіряйте фактичні значення, а не тільки наявність ключів
2. **Timing issues**: Перевіряйте, чи дані встигають зберегтися перед наступними діями
3. **Encoding**: Перевіряйте правильність кодування даних (особливо для спеціальних символів)
4. **Storage events**: Використовуйте `storage` event для відстеження змін
